💡 Finding Your Project Idea — A Framework for Picking What to Build ▾

The best first project is one you actually want to exist. Here's a proven framework for finding it:

1
Find a problem you have personally
What do you do manually that feels annoying or repetitive? What do you Google every week? What spreadsheet do you maintain that wishes it were a proper app? Build the tool that solves your own problem first — you understand the requirements perfectly because you are the user.
2
Stay in scope for your current skill level
For Project 1, you want something that fits in a single page, doesn't need user accounts, and doesn't talk to external services. Good scope: a habit tracker, a reference page for your hobby, a tool to calculate something you calculate often. Bad scope for Project 1: an e-commerce store, a social network, or anything that starts with "like Airbnb but for..."
3
Use Claude to validate your idea before building
"I want to build [your idea]. I'm a complete beginner and this is my first project. Is this a good scope for a single-page app with no database? What are the core features I should include and what should I leave out to keep it simple? What might be harder than I expect?"

🖥️ Terminal Crash Course — Everything You Need to Know ▾

The terminal is just a text-based way to control your computer. Instead of clicking icons, you type commands. That's it. Here are the only commands you'll use in this course:

pwd
Where am I?
pwd prints your current location. Think of it as asking "what folder am I in right now?"
ls
What's in this folder?
ls (Mac/Linux) or dir (Windows) lists everything in your current folder — like opening a folder in Finder or Explorer.
cd
Move to a folder
cd my-dashboard2 moves you into a folder called "my-dashboard2". cd .. goes up one level. cd ~ takes you back home.
mkdir
Create a folder
mkdir my-project creates a new folder called "my-project" in your current location.
Pro tip: Press Tab to auto-complete folder and file names. Press ↑ arrow to repeat your last command. Press Ctrl+C to cancel anything that's running.
File paths demystified: A path is just directions to a file. /Users/yourname/projects/my-dashboard2/index.html means: start at the root, go into Users, then yourname, then projects, then my-dashboard2, and the file is index.html. Every file on your computer has a path like this.

🔧 When Things Break — A Debugging Framework ▾

Most beginners panic when something breaks. Experienced developers follow a process. Here it is:

1
Open the browser console first
Press F12 (Windows) or Cmd+Option+I (Mac), click the Console tab. Red text = errors. This is where 80% of problems announce themselves. Copy the exact error text — don't try to understand it, just copy it.
2
Give Claude Code the full picture
The best bug report includes: what you expected to happen, what actually happened, and the exact error message. Don't say "it's broken" — say "I clicked the login button and expected to be redirected to the dashboard but instead I got this error: [paste error]"
3
Git is your undo button
If Claude Code makes changes that break something badly, you can always go back to a working state:
git log --oneline # see your recent commits git checkout abc1234 -- . # restore to that commit
This is why committing regularly matters — it creates restore points. Commit every time something works.
4
When to simplify instead of debug
If you've spent more than 20 minutes on a bug and Claude Code keeps going in circles, the problem is often scope: too many changes at once. Ask Claude Code: "Let's strip this back to the simplest version that should work, confirm it works, then add back features one at a time."

🗄️ Step 1 — Set Up Supabase (Your Free Database) ▾

What is Supabase? Supabase is a free, open-source backend platform that gives your app three things it needs to work as a real product: a database to store all your users' data, authentication so users can sign up and log in securely, and an API that your React app talks to. Think of it as the engine running behind your app — your frontend is what users see, Supabase is everything happening underneath.

Why is it so important? Without Supabase, every user's dashboard would disappear when they close their browser — you'd have no way to save anything. Supabase is what makes your app real: each user gets their own private data that persists forever, survives device changes, and can only be accessed by them. It's also what powers user accounts — sign up, login, and logout are all handled by Supabase Auth, so you never have to deal with storing passwords yourself.

Is it really free? Yes — Supabase's free tier includes up to 50,000 monthly active users, 500MB of database storage, and unlimited API calls. That's more than enough to build, launch, and grow this app.

1
Create a free Supabase account
Go to supabase.com and sign up. Click New Project, give it a name like "dashboard-creator", choose a region close to you, and set a database password. Free tier includes everything you need. Wait about 2 minutes for the project to finish provisioning.
2
Get your Project URL
In the left sidebar, go to Integrations → Data API. Under API URL you'll see your project URL — it looks like https://xxxxxxxxxxxx.supabase.co. Click Copy and paste it somewhere safe.
3
Get your anon/public key
In the left sidebar click Settings → API Keys. At the top you'll see two tabs — click Legacy anon, service_role API keys. Copy the key labelled anon public. That's your anon key.
Never copy the service_role key — it has full database access and must never go in frontend code or GitHub. Only ever use the anon public key in your app.
🔐 Keeping Secrets Safe — The Most Important Habit in Development ▾

You just got your first real API credentials from Supabase. This is the moment developers lose thousands of dollars — by accidentally pushing keys to a public GitHub repo. Automated bots scan GitHub constantly and will exploit a leaked key within minutes.

1
Always store credentials in a .env file
A .env file holds your API keys and passwords. Your code reads from it, but the file itself never goes to GitHub. Ask Claude Code to set one up:
"Create a .env file for this project with my Supabase URL and anon key, and update the code to read from it instead of hardcoding the values."
2
Always add .env to your .gitignore
A .gitignore file tells Git what to never upload. Ask Claude Code: "Make sure .env and any other sensitive files are in .gitignore so they never get pushed to GitHub."
# .gitignore — these files stay local only .env .env.local .env.production node_modules/
3
Add secrets to Vercel separately
Vercel needs your credentials too — but you add them through the dashboard, not your code. In your Vercel project go to Settings → Environment Variables and add each key there. Never put secrets in your code files.
4
What to do if you accidentally commit a secret
Immediately go to Supabase (or whichever service) and revoke that key, then generate a new one. Don't just delete the commit — the key may already be compromised. Speed matters here.

🎨 CSS & Layout Fundamentals — How to Control How Things Look ▾

You don't need to learn all of CSS. You need to understand the mental model so you can describe visual problems clearly to Claude Code — and understand why it makes the changes it does.

1
Everything is a box
Every element on a webpage — a button, a paragraph, an image — is a rectangular box. CSS controls the size of the box (width, height), the space inside it (padding), the space outside it (margin), and its border (border). Understanding this one concept explains 60% of layout behaviour.
2
Flexbox: the layout tool you'll use most
Adding display: flex to a container puts its children in a row. justify-content: center centers them horizontally. align-items: center centers them vertically. flex-direction: column stacks them vertically. That's 80% of what flexbox does — and 80% of what you'll need.
3
Describing visual problems to Claude Code
Don't say "fix the layout". Say: "The login button is touching the top of the input field — I want 16px of space between them." Or: "The card is 100% width on mobile — it should be narrower with some margin on the sides." Specific, visual, measurable descriptions get better results.
4
Use the browser inspector to investigate
In browser dev tools (F12), hover over any element to see its box, padding, margin, and CSS properties. Right-click any element on your page and choose Inspect. You can even edit CSS live in the browser to experiment before asking Claude Code to make the change permanently.

✅ Testing Your App Before and After Shipping ▾

Most bugs that reach users could have been caught with 15 minutes of systematic testing. Run this checklist both locally before deploying and again on the live URL after deploying.

  • Happy path: Sign up as a new user → build a dashboard → log out → log back in → confirm dashboard saved
  • Error states: Try to sign up with an already-used email. Try logging in with the wrong password. What does the user see?
  • Mobile check: Open Chrome DevTools (F12), click the phone icon to toggle mobile view. Does the layout break at 375px (iPhone) width?
  • Console check: Open the console (F12 → Console). Are there any red errors during normal use?
  • Second browser: Open your app in an incognito/private window. This simulates a fresh user with no cookies or localStorage — a common source of bugs.
  • Live URL test: After deploying, repeat the happy path on the real Vercel URL — environment variable issues often only surface in production.
  • Ask Claude Code: "Review this app and identify any obvious bugs, missing error states, or ways a user might break it."

1
Google SSO — Let users sign in with their Google account
Google SSO (Single Sign-On) means users click "Continue with Google" instead of creating a password. It dramatically increases sign-up rates and removes the friction of forgotten passwords. Supabase handles all the complexity.
Part 1 — Create a Google Cloud project and get your credentials
1. Go to console.cloud.google.com and sign in with your Google account.
2. Click the project dropdown at the top → New Project. Name it "dashboard-creator" and click Create.
3. Click the ☰ hamburger menu (top left) → APIs & Services → OAuth consent screen. Google may redirect you to the new Google Auth Platform page instead — that's the same thing with a new name. You'll see a Get started button in the middle of the screen. Click it.
4. Fill in App name (e.g. "Dashboard Creator") and User support email (your email). Click Next.
5. On the Audience screen select External and click Next.
6. On the Contact Information screen enter your email and click Next.
7. On the Finish screen check the agreement box and click Create. You'll land on the Google Auth Platform dashboard.
8. In the left sidebar click Clients → + Create Client. Set Application type to Web application and name it "Dashboard Creator Web".
9. Under Authorised JavaScript origins click + Add URI and add your Supabase project URL: https://xxxxxxxxxxxx.supabase.co
10. Under Authorised redirect URIs click + Add URI and add: https://xxxxxxxxxxxx.supabase.co/auth/v1/callback — replace xxxxxxxxxxxx with your Supabase project's subdomain.
11. Click Create. A popup shows your Client ID and Client Secret — copy both and keep them safe.
Part 2 — Enable Google in Supabase
1. In your Supabase dashboard go to Authentication → Providers. Find Google in the list and click it.
2. Toggle Enable Sign in with Google on.
3. Paste your Client ID and Client Secret from Part 1 into the fields provided. Click Save.
Part 3 — Add the button to your app
"Add Google SSO to my Dashboard Creator app using Supabase Auth. Add a 'Continue with Google' button to both the sign up and login pages. When clicked it should open Google's login flow and redirect back to the dashboard on success. Keep the existing email/password option as well."
Part 4 — Add your live Vercel URL to Google
After deploying, go back to Google Cloud Console → APIs & Services → Credentials, click your OAuth client, and add your live Vercel URL (e.g. https://dashboard-creator.vercel.app) to both Authorised JavaScript origins and Authorised redirect URIs. Then redeploy once more to pick up the change.
