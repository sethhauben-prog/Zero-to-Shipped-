💰 Understanding API Costs — Before You Build Anything ▾

This is the most important thing to understand before building agents. Unlike web apps that run on free Vercel hosting, agents call APIs every time they run — and those calls cost money. Understanding costs upfront prevents nasty surprises.

1
How Claude API pricing works
You pay per token — roughly per word. Claude Sonnet costs approximately $0.003 per 1,000 input tokens and $0.015 per 1,000 output tokens. A typical agent call with a medium-length prompt and response costs about $0.01–$0.05. That's nearly free for testing, but adds up if your agent runs thousands of times.
2
Estimate your agent's monthly cost before deploying
The formula: (runs per day) × (cost per run) × 30. An agent that runs 24 times per day (hourly) at $0.02 per run = $14.40/month. One that runs every 5 minutes at $0.05 = $432/month. Always calculate this before setting a schedule.
3
Set a spending limit before you start
In the Anthropic Console, go to Billing → Usage Limits and set a monthly hard limit — e.g. $20. This caps your exposure if an agent runs in an unexpected loop. Set it before writing a single line of agent code.
4
Keep prompts short and efficient
Every word in your system prompt costs money, multiplied by every run. Cut anything that isn't necessary. A 2,000-token system prompt running hourly costs 10× more than a 200-token one. Ask Claude to: "Rewrite this system prompt to be as concise as possible while keeping all the important instructions."
✍️ Prompt Engineering for Agents — Different from Chatting ▾

Writing prompts for an autonomous agent is very different from chatting with Claude. In a chat, you can clarify and correct. In an agent, the prompt runs unattended — every ambiguity becomes a bug.

1
Give the agent a clear role and scope
Start every agent system prompt with: what the agent is, what it should do, and — crucially — what it should NOT do. Constraints are as important as instructions.
You are an email response assistant for [Business Name]. Your job: read incoming customer support emails and draft a helpful reply. ONLY respond to questions about our products and services. DO NOT make promises about refunds, timelines, or pricing without checking. If you're unsure how to respond, say "I'll need to check on this" instead of guessing.
2
Specify the output format exactly
Agents pass their output to the next step — a webhook, an email, a database. If the format varies, downstream steps break. Tell the agent exactly what to return: "Respond with only a JSON object with keys: reply_text (string), confidence (1-10), needs_human_review (boolean). No other text."
3
Test edge cases before going live
Before turning on an automated agent, manually test at least 5 edge cases: empty input, very long input, input in another language, rude or nonsensical input, and the exact type of input you expect most. Ask Claude: "What are 5 edge cases that might cause this agent to behave unexpectedly? Let's test each one."
🎯 What You're Building — and What an Agent Actually Is ▾

An agent is an AI that doesn't just answer questions — it takes action. Instead of you typing a prompt and reading a response, an agent watches for a trigger (like a new email or a form submission), processes it using AI, and does something with the result (like sending a reply, updating a spreadsheet, or posting to Slack) — all without you touching a keyboard.

What you'll build: A Zapier workflow that watches your Gmail inbox for emails with the subject "AI Help", feeds the email body to Claude, and sends an intelligent AI-written reply automatically — within seconds of receiving the email.

🤖 Step 2 — Build Your First Zap (Agent Workflow) ▾
1
Create a new Zap in Zapier
In Zapier, click Create Zap. A Zap is a workflow — it has a trigger (what starts it) and one or more actions (what happens next).
2
Set the trigger — New email in Gmail
Search for Gmail as your trigger app. Choose the event "New Email". Connect your Gmail account. Under filters, set Subject Contains to AI Help. This means the Zap only fires when you receive an email with that subject line.
3
Add Claude as the action — generate a reply
Click the + to add an action. Search for Anthropic (Claude). Choose "Send Message". Connect using your API key. In the prompt field, enter:
You are a helpful assistant. Someone emailed asking for help. Read their email and write a friendly, helpful reply in 2-3 sentences. Email subject: [insert Gmail subject field here] Email body: [insert Gmail body field here] Write only the reply text — no subject line, no greeting like "Dear", just the reply body.
In the prompt, click into the subject and body fields and use Zapier's variable picker to insert the actual Gmail data from the trigger step.
4
Add a second action — send the reply
Add another Gmail action. Choose "Send Email". Set To to the sender's email from the trigger. Set Subject to Re: [original subject]. Set Body to the output from Claude in the previous step.
5
Test and turn on your agent
Click Test in Zapier to run the workflow with a real email. Review the output — does Claude's reply make sense? Adjust your prompt if needed. When satisfied, toggle the Zap to On. Your agent is now live.
Test it: Send yourself an email with the subject "AI Help" and a question in the body. Within 30 seconds you should receive an AI-written reply.
🔧 Step 3 — Customize and Extend ▾

Now that your basic agent works, try these improvements:

  • Change the trigger — instead of Gmail, try a Typeform submission, a Slack message, or a new row in Google Sheets
  • Change the action — instead of replying by email, post Claude's response to a Slack channel or save it to a Google Doc
  • Improve the prompt — give Claude a persona: "You are a customer support agent for [your business]. Always be friendly, concise, and offer one next step."
  • Add a filter step — only trigger the agent if the email contains certain keywords, or only during business hours
  • Add logging — add a Google Sheets action that saves every email + Claude's reply to a spreadsheet so you can review them

🛡️ Error Handling — Agents Fail Silently Without This ▾

An agent running unattended can fail for hours without anyone noticing. The web search returns nothing. The email sends to the wrong address. Claude returns malformed output. Without error handling, you'll never know. With it, failures surface immediately.

1
Wrap every external call in try/except
Every call to a web API, Claude, or an email server can fail. Ask Claude Code: "Add try/except error handling around every external API call. If a call fails, log the error with a timestamp and continue rather than crashing the whole agent."
2
Validate Claude's output before using it
When your agent expects Claude to return JSON or a specific format, always validate it before passing it downstream. Ask Claude Code: "Before using Claude's response, check that it matches the expected format. If it doesn't, retry once with a clarifying prompt. If the second attempt also fails, log the bad output and skip this run."
3
Add a dead agent alert
Ask Claude Code: "Add a check: if the agent hasn't successfully completed a run in 3 hours, send me an alert email. This catches situations where the agent crashes silently or the server goes down." This one feature will save you countless headaches.
4
Handle empty or poor search results gracefully
What should your research agent do when the search returns no results? Skip and log it? Send an email saying "nothing found today"? Never send Claude an empty results set — it will hallucinate content that doesn't exist. Always check: "If the search returns fewer than 2 results, skip this run and log 'insufficient results' rather than generating a digest."

🔌 MCP Servers — Give Your Agent Native Tool Access ▾

The Model Context Protocol (MCP) is a standard that lets Claude connect directly to external tools and data sources — without you writing custom API integration code. Instead of building a custom Slack connector, you install a Slack MCP server and Claude can read and write to Slack natively.

1
What MCP unlocks for agents
With MCP servers, your agent can connect to Google Drive, GitHub, Slack, Notion, databases, and dozens of other services — using pre-built connectors instead of custom code. This dramatically reduces the work needed to build powerful agents.
2
Finding and installing MCP servers
Browse available MCP servers at the official registry. To add one to your project, ask Claude Code:
"I want to connect my agent to [Notion / GitHub / Slack / Google Drive]. Find the appropriate MCP server for this, install it, and configure it so my agent can read and write to it. Walk me through getting the required API credentials."

Add MCP memory to your memory agent

Ask Claude Code: "Connect my memory agent to my Google Calendar via MCP. When I ask 'what do I have coming up?', the agent should check my actual calendar and factor it into its response."

📊 Evaluating Your Agent — How to Know If It's Actually Working ▾

A deployed agent that's quietly making bad decisions is worse than a broken one. Production agents need evaluation — a way to measure whether they're doing their job well, catching when quality degrades, and knowing when something has changed.

1
Define what "good" looks like before you deploy
For each action your agent takes, decide upfront what success looks like. For a research agent: "at least 3 unique sources, all from the last 7 days, no repeated topics from yesterday's digest." Write these criteria into your code as assertions — if the agent's output doesn't meet them, it logs a quality warning.
2
Build a review log you actually read
Ask Claude Code: "Add a weekly summary email that shows me: total runs this week, success rate, number of actions taken, any quality warnings triggered, and 3 examples of the agent's output. Send it every Monday morning." An agent you never inspect is one you can't improve.
3
Detect when the agent's quality has changed
LLM outputs can degrade over time as context grows, prompts drift, or the underlying model changes. Ask Claude Code: "Add a quality score to each agent run — ask Claude to self-evaluate its own output on a 1-10 scale. Log this score over time and alert me if the 7-day average drops below 7."
🔒 Trust Boundaries — What Your Agent Should Never Be Allowed to Do ▾

An autonomous agent that can take action in the real world needs hard limits on what it can do — not soft guidelines, but code-enforced guardrails. Trust boundaries are the difference between a useful tool and an expensive mistake.

1
The principle of least privilege
Give your agent access to only what it needs for its specific job — nothing more. If it only needs to read emails, don't give it permission to send them. If it only posts to one Slack channel, don't give it access to all channels. Scope credentials tightly and document what each one is for.
2
Hard-code irreversible action limits
Some actions can't be undone — sending an email, posting publicly, deleting data, spending money. For each irreversible action, add a code-level rate limit that the agent cannot override regardless of what its LLM reasoning says. Ask Claude Code: "Add hard rate limits: maximum 10 emails per hour, maximum 50 Slack messages per day. If either limit is hit, stop all actions and send me an alert — do not allow the agent to bypass this."
3
Build a human-in-the-loop escape hatch
For high-stakes decisions, don't let the agent act alone. Ask Claude Code: "For any action with an importance score above 8, instead of taking the action automatically, send me an approval request via email or Slack with a one-click approve/reject link. Only proceed if I approve within 24 hours."
